# Gemini 内容提炼：适配 ARSM 的需求清单

日期：2026-06-21
用途：把 Gemini 给出的 Python/Flet/本地播放器方案，转换成适合当前 ARSM（Next.js + TypeScript + Prisma + SQLite + Web）的可执行需求池。

---

## 一、先说结论

Gemini 那份方案不能直接作为 ARSM 的开发路线，因为它默认的是：
- Python + Flet
- VLC 本地播放器内核
- 绿色单机 `.exe`
- 本地媒体库程序

而 ARSM 当前已经是：
- Next.js + TypeScript + Prisma + SQLite
- Web 作品库 / 管理后台 / 多用户结构
- 采集 / 导入 / 元数据 / 播放 / 字幕 / 统计

因此：

**不能照抄技术栈，不能重开 Python/Flet 新项目。**

但其中有不少“产品能力”和“参考方向”适合被吸收进 ARSM 的后续阶段。

---

## 二、可直接吸收进 ARSM 的需求

### A. 元数据补全增强

Gemini 里最有价值的一类，是更完整的作品元数据。

适配到 ARSM 后，可拆成这些需求：
- 作品补全更多字段：封面、社团、CV、发售日、标签、价格、来源链接
- 多来源元数据融合：asmr.one + DLsite 或其他来源
- 元数据重抓策略：已有字段不覆盖 / 空字段自动补齐 / 管理员强制刷新
- `workSources` 的前端可见化：展示作品有哪些来源、哪条来源更完整

建议优先级：**高**

---

### B. 高级搜索能力

Gemini 提到的命令式搜索语法很适合移植成 ARSM 的增强搜索。

适配到 ARSM 后，可以做：
- 关键词 + 排除词
- 多标签条件搜索
- 时长区间搜索
- 是否有字幕 / 是否收藏 / 是否已评分 的筛选
- 社团 / CV / 标签 联合过滤

进一步可选：
- 高级搜索语法，例如：
  - `护士 -中出`
  - `cv:秋野花 tag:舔耳`
  - `duration>60m`

建议优先级：**高**

---

### C. 本地音频文件标签写入

Gemini 提到 `mutagen` 这类思路，本质上是：

**把 ARSM 已经抓到的元数据，反写进本地音频文件本身。**

适配到 ARSM 后，可以做成可选工具：
- 将标题写入音频标签
- 将社团 / CV / 专辑名写入标签
- 将封面图嵌入音频文件
- 对指定作品执行“整理并写回标签”

注意：
- 这是“文件改写”能力，风险比普通浏览高
- 应该做成手动触发，不要默认批量改写

建议优先级：**中**

---

### D. 播放器增强（适合后续阶段）

Gemini 那份本地播放器思路里，有些功能对 ASMR 很实用。

适配到 ARSM 后，可考虑做：
- 左右声道平衡控制
- 单声道合并 / 单耳模式
- 睡眠定时关闭
- 更细的倍速与播放模式
- 播放器设置持久化

注意：
- 这类功能属于播放器增强，不该打断当前来源 / 元数据 / 统计主线
- 适合在 **Phase 10 或播放器增强专题** 中做

建议优先级：**中低**

---

### E. 字幕与文本生态增强

Gemini 提到 AI 字幕、字幕文本等方向，这和 ARSM 现有字幕链路是兼容的。

适配到 ARSM 后，可拆成：
- 字幕文件管理增强：原文 / 中文字幕 / 台本分组
- 双语字幕显示
- 文本搜索命中到字幕内容
- AI 翻译字幕（后期）
- AI 生成字幕 / 转写（更后期）

建议优先级：
- 双语/文本管理：**中**
- AI 字幕翻译 / 转写：**低，后期**

---

### F. 本地下载与归档结构优化

Gemini 对“保持多级目录结构”这一点是对的，这一点已经和 ARSM 的导入/分组方向一致。

适配到 ARSM 后，可继续强化：
- 导入后保留更完整的目录语义
- 区分 `本篇 / SE有 / SE无 / WAV / MP3 / 特典 / 台本`
- 在作品详情页更清楚地展示目录层级
- 为不同组提供单独播放入口或过滤按钮

建议优先级：**中高**

---

## 三、可以参考但不能直接照抄的方向

### A. 参考开源项目的“逻辑”，不要照搬架构

Gemini 列出的开源项目，可以拿来做这几类参考：
- asmr.one API 请求封装
- RJ 元数据解析逻辑
- 标签写回音频文件
- 下载任务结构
- 本地媒体库组织方式

但不要这样做：
- 不要按它们的语言 / GUI 框架重写 ARSM
- 不要把 CLI 工具直接搬成 ARSM 主架构
- 不要为了某个单点能力打断现有 Web 项目主线

---

### B. 可以让 DeepSeek 去“调研式抄逻辑”

推荐给 DeepSeek 的方式是：
- 只抽 API 请求层
- 只抽元数据解析层
- 只抽标签写入层
- 只抽搜索语法层

不要让它：
- 整个项目照抄
- 直接重构为 Python/Flet/VLC 单机程序

---

## 四、不建议现在做的事

这些内容来自 Gemini，但当前不适合进入 ARSM 主线：
- 重开 Python + Flet 新项目
- 把 ARSM 改造成单机绿色播放器主架构
- 用 VLC 重写现有 Web 播放器主线
- 为了桌面壳而打断来源 / 元数据 / 统计 / 验收主线

结论：

**这些可以作为“未来桌面壳/本地版产品探索”，不能作为当前 ARSM 主线。**

---

## 五、建议并入 ARSM 的阶段归属

### 适合并入当前 / 下一阶段（Phase 9）
- 第二真实元数据来源深化
- DLsite 元数据补全增强
- 高级搜索能力
- 更完整的个人统计页
- 后台更清楚的来源能力边界提示

### 适合并入后续阶段（Phase 10）
- 声道平衡 / 单声道模式
- 睡眠定时关闭
- 音频标签写回文件
- 双语字幕展示

### 适合远期探索（Phase 11+）
- AI 字幕翻译
- AI 自动转写
- 桌面绿色壳 / 本地单机壳
- Android 客户端

## 六、给 DeepSeek 的执行提示

如果要把这份清单交给 DeepSeek，建议直接这样说：

1. 不要改技术栈，继续使用当前 ARSM 栈。
2. 不要重开 Python/Flet 新项目。
3. 只吸收 Gemini 内容中“适合 ARSM 的能力需求”。
4. 优先做：
   - 元数据来源深化
   - 高级搜索
   - 统计面板深化
   - 来源边界提示
5. 暂时不要做：
   - 桌面绿色版重构
   - AI 字幕
   - VLC 本地播放器主线替换
