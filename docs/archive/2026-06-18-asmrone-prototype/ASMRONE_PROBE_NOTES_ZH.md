# asmr.one 资源探测实测备注

## 本次结论

这次已经不是纸上分析，而是实测跑通到了“资源枚举”阶段。

已确认：

- 作品页真实走的是 `api.asmr-200.com`
- 页面会请求：
  - `/api/workInfo/{id}`
  - `/api/tracks/{id}?v=2`
  - `/api/work/{id}`
- 点下载后会弹出站内文件树
- 文件树里已经能枚举出完整目录结构与文件名
- 第二次点下载后走的是浏览器目录选择，而不是普通单文件另存为

所以当前最稳的技术判断是：

- “爬资源”已经可以做到文件清单级别
- “自动下载”还差最后一段真实文件请求/保存逻辑

## 本次最重要的突破

以前只是知道页面上有“下载”按钮。

现在已经知道：

1. 真实 API 域名
2. 真实 work / tracks 接口
3. 下载弹窗文件树确实能被自动化读取
4. 文件树结构足够拿来做 dry-run inspector

这意味着后续 provider 已经可以先做出来：

- 输入 `RJ01538000`
- 自动打开作品
- 自动展开下载弹窗
- 自动读取文件树
- 输出资源清单 JSON

这一步已经足以作为 ARSM 里的第一版“采集探针”。

## 还没完全跑通的最后一段

还没最终抓到的是：

- 每个文件真正下载时的请求 URL
- token 如何带上
- 浏览器目录写入到底是前端逐文件写，还是后端先生成临时文件对象

## 因此建议的开发边界

第一版 provider 先做到：

- `inspect()` 可用
- 输出完整目录树
- 输出文件类型分布
- 输出总大小
- 标记“可见但尚未自动下载”

第二版再补：

- 真实逐文件下载
- 目标仓库写入
- 导入 ARSM
